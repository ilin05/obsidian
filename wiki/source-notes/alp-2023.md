# ALP: Adaptive Lossless floating-Point Compression

Afroozeh, Kuffo, and Boncz, SIGMOD 2023 (CWI Amsterdam).

## Summary

ALP takes a fundamentally different approach from the XOR-based streaming lineage. It observes from a study of 30 datasets that (1) many floating-point values were originally generated as decimals (monetary amounts, coordinates, temperatures), and (2) even for non-decimal values, the exponent and high mantissa bits are highly similar within a vector. ALP is **adaptive per 1024-value vector**: if the vector can be exactly encoded as scaled integers (enhanced PseudoDecimals), it does so; otherwise, it compresses only the front bits (sign + exponent + high fraction bits). Critically, ALP is **vectorized** — designed for SIMD execution in auto-vectorizing C++, achieving 1–2 orders of magnitude faster [de]compression than Chimp/Elf while matching or exceeding their compression ratios.

## Key Contributions

- **Dataset analysis (30 datasets)**: Systematic study of decimal precision, exponent deviation, leading/trailing zero patterns, and value similarity — both for time series and non-time-series data. Found that per-vector commonalities are strong even without temporal ordering.
- **Two-strategy adaptive design**:
  - **ALP RD (Real Decimals)**: For vectors where all values can be exactly represented as integers after scaling by a common factor (e.g., all values have ≤2 decimal digits → multiply by 100). Uses vectorized bit-packing of the resulting integers.
  - **ALP Front-Bits**: For vectors where decimal encoding fails, stores only the first N bits of each double (sign + exponent + top fraction bits), where N varies per vector. The remaining bits are stored separately, exploiting high similarity in the front part.
- **Two-level sampling**: Row-group level determines whether decimals are present; vector level picks the best encoding per vector. Amortizes adaptivity overhead — no per-value control bits.
- **Vectorized execution**: Encoding/decoding operates on 1024-value arrays using auto-vectorizable scalar C++. Per-vector branching (not per-value) avoids branch mispredictions.
- **Cascade compression**: ALP output can be further compressed with lightweight encodings (Dictionary, Delta, RLE, FOR) using the FastLanes library.

## Reported Results

- [De]compression speed: **10–100× faster** than Chimp128, Elf, and Gorilla
- Compression ratio: similar to Zstd, better than all other streaming float codecs on most datasets
- On decimal-originated data (14 of 30 datasets): ALP RD achieves 2–4× compression with essentially zero decode overhead
- On non-decimal data: ALP Front-Bits compresses to ~50–70% of original size
- Two-stage sampling adds <1% overhead compared to optimal per-vector oracle selection

## Significance

ALP demonstrated that the XOR-based streaming approach (Gorilla → Chimp → Elf) is not the only path — **vectorized adaptive encoding** can simultaneously improve both speed and ratio. Its key insight — that many "floating-point" values are really decimals stored in float format — is widely applicable. The vectorized design breaks from the per-value state machine paradigm of all prior float codecs, trading streaming capability for SIMD throughput. This makes ALP particularly well-suited for columnar database systems (DuckDB, ClickHouse) and big data formats (Parquet) where vector-at-a-time processing is standard.

## Limits And Open Questions

- Not streaming: requires 1024-value vectors; inapplicable to single-value-at-a-time scenarios (e.g., real-time sensor ingestion without batching).
- The decimal detection threshold is dataset-dependent; the current sampling approach may misclassify vectors at boundaries.
- Front-bits encoding loses trailing mantissa bits, which must be stored uncompressed — on high-entropy data, this can negate gains.
- The vectorized design targets CPU SIMD; GPU or FPGA acceleration is not explored.
