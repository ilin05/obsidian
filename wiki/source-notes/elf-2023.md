# Elf: Erasing-based Lossless Floating-Point Compression

Li, Li, Wu, Chen, and Zheng, PVLDB 2023 (Chongqing University / JD Technology).

## Summary

Elf observes that the key bottleneck in XOR-based float compression is the **scarcity of trailing zeros** in XOR results — 95% of XORed values have ≤5 trailing zeros with Gorilla/Chimp, and even Chimp128 leaves ~60% in that range. Elf introduces an **erasing-based** approach: deliberately zero out the last k bits of the current float value before XORing with the reference. This transforms the XOR result to have many more trailing zeros (44 in the paper's running example vs 2 without erasing). A mathematical framework determines the erased bit count in O(1) and guarantees lossless recovery during decompression by adding back a bounded error term. Elf achieves 12.4% better compression than Chimp128 and 43.9% better than Gorilla on average across 22 datasets, while remaining streaming (O(N) time, O(1) space).

## Key Contributions

- **Erasing strategy**: Systematically zero low-order mantissa bits of the current value before XOR, creating long trailing zero runs in the XOR result. The erased value is always smaller than the original, and the gap is bounded.
- **O(1) eraser**: Uses the decimal significand count of the current value to determine how many bits can be safely erased without ambiguity. The bound depends on the difference between the current and reference values' decimal scales.
- **Lossless restorer**: During decompression, recovers the original value by adding Δ = (original − erased) to the decompressed erased value. Since Δ is bounded by a known ε, and the original is the unique value within [erased, erased+ε) whose decimal representation matches the stored metadata, recovery is exact.
- **Elf XOR compressor**: A specialized XOR encoding optimized for the post-erasure distribution — many XOR results now have 40+ trailing zeros, so fewer center bits need storing.
- **Streaming design**: Elf Compressor (Eraser → XORcmp) and Elf Decompressor (XORdcmp → Restorer) run in a single streaming pass.

## Algorithm Sketch

```
Compress:
1. Eraser: given current v_curr and reference v_ref, determine k
   erased bits such that v_curr' = truncate_trailing(v_curr, k)
   has maximal trailing zeros without losing recoverability.
2. XORcmp: encode v_curr' ⊕ v_ref using trailing-zero-rich encoding.

Decompress:
1. XORdcmp: decode v_curr' from the bitstream using v_ref.
2. Restorer: recover v_curr = v_curr' + δ, where δ ∈ (0, ε)
   and v_curr is uniquely determined by its decimal representation.
```

## Reported Results

- 12.4% average improvement over Chimp128, 43.9% over Gorilla (22 datasets)
- Competitive with general-purpose Xz on compression ratio, but ~20× faster
- Particularly strong on datasets with low-to-medium decimal precision (≤5 decimal digits)
- Compression and decompression speed are slower than Chimp due to erasing/restoring overhead (~2–3× slower than Chimp, but still ~5× faster than Xz)
- On high-precision datasets (>10 decimal digits), gains diminish because fewer bits can be safely erased

## Significance

Elf is the first method to **actively manipulate** the float representation to improve XOR compressibility, rather than passively exploiting existing trailing zeros. This opened a new design dimension: pre-XOR transformations that increase trailing zero counts. The erasing approach inspired subsequent decimal-domain methods (DeXOR) that achieve similar effects through common prefix extraction rather than bit erasure.

## Limits And Open Questions

- Erasing and restoring add compute overhead — Elf is 2–3× slower than Chimp in compression speed.
- The erasing bound depends on decimal precision; high-precision values (>10 decimal digits) allow few bits to be erased, limiting gains.
- The mathematical framework assumes the reference value's decimal properties are known; interaction with multi-predecessor search is not explored.
- On integer-dominated datasets, the benefit over simpler integer encoding (ALP, PDE) is unclear.
