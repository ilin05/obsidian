# QJL: 1-Bit Quantized JL Transform for KV Cache Quantization with Zero Overhead

## Summary

QJL proposes a KV-cache quantization method built around a Johnson-Lindenstrauss transform followed by sign-bit quantization. Its central claim is that it removes the usual per-block quantization overhead because it does not need stored scale or zero-point constants, while still preserving attention-relevant inner products well enough for inference.

## Key Contributions

- Introduces a quantized JL sketch for key embeddings that uses 1-bit sign quantization after random projection.
- Derives an asymmetric inner-product estimator: the key is quantized, the query is projected but not quantized.
- Claims zero quantization-constant overhead relative to blockwise schemes such as per-channel or per-token affine quantization.
- Adds a practical CUDA-oriented implementation and an outlier-handling split between inlier and outlier channels in deeper layers.

## Reported Results

- Quantizing KV cache values to 3 bits per floating-point number reportedly gives more than a 5x KV-cache memory reduction.
- The paper reports no accuracy drop versus full precision on the tested long-context tasks and faster runtime than the baseline for long contexts.
- It also claims stronger LongBench results than recent KV-cache quantization baselines while reducing memory overhead.

## Why It Matters

- It reframes KV-cache quantization as an inner-product-preservation problem rather than only an MSE minimization problem.
- Its zero-overhead framing is important because metadata overhead can erase much of the benefit of low-bit quantization.
- It is a direct precursor to [TurboQuant](../entities/turboquant.md), which reuses QJL on residuals after a stronger MSE-oriented first stage.

## Limits And Open Questions

- The paper is specialized to KV-cache style workloads and does not by itself give a general-purpose vector quantization framework.
- The outlier handling is practical and important, but it suggests that the clean theoretical sketch still needs model-aware adjustments in deeper layers.
- It is still unclear from this source alone when plain QJL is preferable to the more general two-stage TurboQuant pipeline.
